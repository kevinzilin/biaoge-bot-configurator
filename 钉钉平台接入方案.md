  三、详细改造清单

  步骤 1：完善抽象接口（ports.py 扩展）

  在现有 ports.py 中补充缺失的接口定义：

  # AuthPort - 认证
  class AuthPort(Protocol):
      async def get_token(self) -> str: ...

  # IMPort - 消息收发
  class IMPort(Protocol):
      async def send_text(self, *, target: str, target_type: str, text: str) -> None: ...
      async def send_image(self, *, target: str, target_type: str, image_key: str) -> None: ...
      async def send_file(self, *, target: str, target_type: str, file_key: str) -> None: ...
      async def send_card(self, *, target: str, target_type: str, card: dict) -> None: ...
      async def upload_image(self, *, file_path: str) -> str: ...
      async def upload_file(self, *, file_path: str) -> str: ...
      async def download_media(self, *, media_key: str, save_path: str) -> None: ...

  # EventPort - 事件订阅与连接
  class EventPort(Protocol):
      def start(self) -> None: ...
      def on_message(self, handler: Callable) -> None: ...
      def on_card_action(self, handler: Callable) -> None: ...
      def on_table_event(self, handler: Callable) -> None: ...

  # CardPort - 卡片构建（平台相关 JSON Schema）
  class CardPort(Protocol):
      def build_panel_card(self, ...) -> dict: ...

  步骤 2：创建飞书适配器（adapters/feishu/）

  将现有平台相关代码迁移到适配器目录，不改变行为：

  步骤 4：钉钉多维表方案选择（关键决策）

  这是整个方案中风险最高的一环。查到的信息显示钉钉多维表的 REST API 文档不完整。有三种实现路径：

  A. 钉钉多维表 REST API（理想路径）

  如果 API 文档足够，直接对接：
  GET  /v1.0/aitable/apps/{appId}/tables/{tableId}/records       # 查询记录（支持筛选）
  POST /v1.0/aitable/apps/{appId}/tables/{tableId}/records       # 新增记录
  PUT  /v1.0/aitable/apps/{appId}/tables/{tableId}/records/{id}  # 更新记录
  GET  /v1.0/aitable/apps/{appId}/tables/{tableId}/fields        # 字段列表

  B. 宜搭（Yida）低代码平台 API（备选路径）

  宜搭表单有成熟的 REST API，可以用"表单"替代"多维表格"：
  - 表单记录增删改查 API 完善
  - 支持附件字段
  - 支持流程状态流转
  - 缺点：不是表格形态，UI 不同

  C. 本地数据库 + 群消息闭环（保底路径）

  如果云端 API 都不可用，用 SQLite 做后端：
  - 状态管理在本地数据库
  - 通过群消息发送"控制面板卡片"来操作
  - 附件通过群文件传递
  - 缺点：不支持多人协同编辑表格

  建议：在阶段 1（调研）中优先验证路径 A，同时准备好路径 C 作为保底。

  步骤 5：改造 AppContext

  让 AppContext 持有平台适配器引用：

  @dataclass(frozen=True)
  class AppContext:
      # ... 现有字段 ...
      platform: PlatformAdapter  # 新增：平台适配器

  @dataclass(frozen=True)
  class PlatformAdapter:
      auth: AuthPort
      im: IMPort
      table: TablePort | None      # None = table mode off
      drive: DrivePort | None      # None = drive off
      event: EventPort
      card: CardPort
      bitable_trigger: Any | None  # trigger handler
      bitable_writeback: Any | None

  步骤 6：改造核心模块（最小化改动）

  main.py：

  改动点 → 根据 PLATFORM 环境变量选择适配器：
  def main():
      platform = os.environ.get("PLATFORM", "feishu")
      if platform == "dingtalk":
          adapter = create_dingtalk_adapter()
      else:
          adapter = create_feishu_adapter()
      ctx = build_context(adapter)
      adapter.event.start()  # 平台统一启动方式

  dispatcher.py：

  改动点 → 将飞书特有调用通过 ctx.platform 转发：
  - im.send_text(chat_id=..., text=...) → ctx.platform.im.send_text(target=chat_id, target_type="chat_id", text=...)
  - 直接 import _enc_* → 通过 ctx.platform.table_logic.xxx() 调用
  - build_panel_card() → ctx.platform.card.build_panel_card()

  callback_server.py：

  改动点 → 回写调用通过 ctx.platform：
  - _enc_write_back_record() → ctx.platform.table_writeback.write_back_record()
